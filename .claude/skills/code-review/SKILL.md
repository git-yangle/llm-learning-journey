## Additional Resources

- 文件过滤规则：[checklist.md](checklist.md)

## 执行步骤

### 步骤一：确定 review 范围

根据 `$ARGUMENTS` 的值判断 review 模式：

- **无参数**：执行 `git log` 找出最近两次 `git push` 之间的提交，提取变更文件列表，按 checklist.md 过滤后进行 review
- **文件路径**：
  - 若文件不存在，告知用户路径有误并退出
  - 若文件命中 checklist.md 排除规则（如 `_test.go`、`_test.py`），提示"该文件不在 review 范围内，是否仍要继续？"，等待用户确认
  - 否则直接对该文件进行 review
- **MR/PR 链接**：使用 `gh pr view` 获取差异内容；若链接无法访问，告知用户并退出

### 步骤二：执行 review

对每个文件从以下维度逐一检查：

| 维度 | 检查内容 |
|------|----------|
| 安全性 | SQL 注入、XSS、硬编码密钥、不安全的反序列化、权限校验缺失 |
| 正确性 | 逻辑错误、边界条件处理、空指针/空值处理、并发安全 |
| 代码风格 | 命名规范、注释完整性、是否符合项目 CLAUDE.md 编码规范 |
| 性能 | N+1 查询、不必要的循环嵌套、资源泄漏（连接/文件未关闭） |
| 可维护性 | 函数过长、重复代码、魔法数字、单一职责原则 |

### 步骤三：向用户展示概要并确认

分析完成后，输出概要并询问用户：

> review 完成，共检视 N 个文件，发现 🔴 严重 X 个、🟡 建议 Y 个、💡 优化 Z 个。
> 是否将完整报告写入项目根目录？

等待用户确认后，执行步骤四。

### 步骤四：写入 review 报告

将完整报告生成到当前项目根目录，文件名格式：`{YYYYMMDDHHmmss}-code-review.md`
